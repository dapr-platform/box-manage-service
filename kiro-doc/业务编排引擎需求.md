# 业务编排引擎需求文档

## 1. 项目概述

### 1.1 项目背景
在现有的 AI 盒子管理系统（box-manage-service）基础上，增加业务编排引擎功能模块，为用户提供可视化的业务流程编排能力。通过拖拽式的流程设计，用户可以灵活组合各种业务节点，实现复杂的业务逻辑自动化。

### 1.2 项目目标
- 提供灵活的业务流程编排能力
- 支持可视化流程设计和配置
- 实现流程的分布式执行（服务端配置，盒子端执行）
- 支持多种触发方式（事件触发、定时触发）
- 提供完整的流程监控和日志追踪

### 1.3 技术架构
- **开发方式**: 前后端分离，本文档聚焦后端功能实现
- **集成方式**: 在现有 box-manage-service 项目框架中增加功能模块
- **部署模式**: 服务端配置 + 盒子端执行
- **技术栈**: Go 1.23.1 + PostgreSQL + GORM + Chi Router

## 2. 核心概念

### 2.1 工作流（Workflow）
工作流是一个完整的业务流程定义，包含流程结构、节点配置、连接关系等信息。

**特性**:
- 一个工作流可以被多次执行
- 每次执行产生一个独立的工作流实例（WorkflowInstance）
- 支持版本管理和历史记录
- 流程结构以 JSON 格式存储

### 2.2 工作流实例（WorkflowInstance）
工作流实例是工作流的一次具体执行，记录本次执行的所有运行时数据。

**特性**:
- 记录执行状态、开始时间、结束时间
- 关联执行日志（WorkflowLog）
- 记录执行上下文和变量值
- 支持暂停、恢复、终止操作

### 2.3 节点（Node）
节点是工作流中的基本执行单元，承载具体的业务逻辑或流程控制逻辑。

**节点分类**:

#### 2.3.1 逻辑控制类节点
- **Start（开始节点）**: 流程入口，每个工作流有且仅有一个开始节点
- **End（结束节点）**: 流程出口，每个工作流有且仅有一个结束节点，所有分支汇聚于一个 end 节点
- **ConcurrencyStart（并发开始）**: 标记并发执行区域的开始
- **ConcurrencyEnd（并发结束）**: 标记并发执行区域的结束，等待所有并发分支完成
- **LoopStart（循环开始）**: 标记循环区域的开始
- **LoopEnd（循环结束）**: 标记循环区域的结束，判断是否继续循环

#### 2.3.2 业务执行类节点
- **KVM 接入节点**: 连接和控制 KVM 设备
- **Reasoning 推理节点**: 调用 AI 模型进行推理计算
- **PythonScript 脚本节点**: 执行自定义 Python 脚本
- **MQTT 推送节点**: 向 MQTT 服务器推送消息


### 2.4 变量（Variable）
变量用于在节点之间传递数据，支持节点的输入输出。

**变量类型**:
- **入参（Input）**: 节点执行所需的输入数据
- **出参（Output）**: 节点执行产生的输出数据

**变量作用域**:
- **全局变量**: 整个工作流实例共享，start 节点的入参为全局变量，后续所有节点均可以选择使用
- **节点变量**: 仅在当前节点及其子节点可见

### 2.5 连接线（Line）
连接线用于连接节点，定义流程的执行顺序和流转路径。

**特性**:
- 连接源节点（Source）和目标节点（Target）
- 可以添加条件（Condition）实现分支逻辑
- 支持条件表达式（如：`age > 18`, `status == "success"`）

### 2.6 条件（Condition）
条件用于控制流程的分支走向，基于变量值进行判断。

**条件类型**:
- **无条件（none）**: 连接线无条件执行
- **简单条件（simple）**: 单个变量的比较（等于、大于、小于等）
- **复合条件（complex）**: 多个条件的逻辑组合，支持 AND/OR 逻辑
- **表达式条件（expression）**: 支持复杂的表达式计算

**逻辑类型（用于复合条件）**:
- **AND 逻辑**: 所有条件都为真时，连接线才执行
- **OR 逻辑**: 任意一个条件为真时，连接线就执行

**复合条件示例**:
```json
// AND 逻辑：所有条件都必须满足
{
  "condition_type": "complex",
  "logic_type": "and",
  "condition_expression": "result.confidence > 0.8; result.status == 'success'"
}

// OR 逻辑：任意条件满足即可
{
  "condition_type": "complex",
  "logic_type": "or",
  "condition_expression": "result.error != ''; result.timeout == true"
}
```

**表达式语法**:
- 支持比较运算符：`==`, `!=`, `>`, `<`, `>=`, `<=`
- 支持逻辑运算符：`&&`, `||`, `!`
- 支持变量引用：`node_key.output_key`
- 支持嵌套属性访问：`node_key.output_key.property`

### 2.7 部署对象（Deployment）
部署对象是 workflow 的一层封装，具有独立的标识和配置，支持同一 workflow 在同一盒子上运行多个实例。

**特性**:
- 每个部署有独立的名称（name）、标识（key）和描述（description）
- 一个 workflow 可以创建多个部署实例
- 每个部署实例可以配置不同的参数（通过 variable_instances 关联）
- 支持在同一盒子上运行同一 workflow 的多个实例

**架构关系**:
```
Workflow (流程定义)
    ↓ 1:N
Deployment (部署实例)
    ↓ N:N (通过 Schedule 关联)
Schedule (调度配置)
    ↓ 1:N
ScheduleInstance (调度实例)
    ↓ 1:N
WorkflowInstance (执行实例)
```

**使用场景**:
- 同一视频分析 workflow 处理多个摄像头
- 同一数据处理 workflow 处理多个数据源
- 同一监控 workflow 监控多个设备

**优势**:
- **Workflow 复用**: 一个 workflow 定义可以被多次部署
- **参数隔离**: 不同部署实例的参数互不干扰
- **灵活调度**: 每个部署可以有独立的调度策略，或多个部署共享一个调度
- **易于管理**: 通过 name 和 key 快速识别部署实例

### 2.8 调度配置（Schedule）
调度配置定义了如何触发部署实例的执行，支持配置多个部署实例。

**特性**:
- 一个调度可以配置多个部署实例（通过 deployment_ids 字段）
- 支持定时触发（cron）和手动触发（manual）
- 每次触发会创建一个调度实例（ScheduleInstance）
- 调度实例会为每个配置的部署创建对应的工作流实例

**调度类型**:
- **manual（手动触发）**: 用户手动启动调度
- **cron（定时触发）**: 按照 Cron 表达式定时执行
- **event（事件触发）**: 基于系统事件触发（暂不实现）

**使用场景**:
```
调度: 每小时视频分析
- deployment_ids: [1, 2, 3]
  - 部署1: video_analysis_camera1 (摄像头1)
  - 部署2: video_analysis_camera2 (摄像头2)
  - 部署3: video_analysis_camera3 (摄像头3)
- 每小时触发一次，同时分析三个摄像头的视频
```

**优势**:
- **批量执行**: 一次调度触发多个部署实例
- **统一管理**: 多个部署共享同一个调度策略
- **灵活配置**: 可以动态调整调度关联的部署列表

### 2.9 调度实例（ScheduleInstance）
调度实例记录每次调度触发的实际数据，包括触发时间、触发的部署列表、创建的工作流实例等。

**特性**:
- 每次调度触发时创建一个调度实例
- 记录触发时间、触发类型、触发数据
- 记录本次触发创建的所有工作流实例
- 记录执行结果（成功数、失败数）

**状态**:
- **pending**: 等待执行
- **running**: 执行中（至少有一个工作流实例在运行）
- **completed**: 已完成（所有工作流实例都已完成）
- **failed**: 执行失败（所有工作流实例都失败）
- **cancelled**: 已取消

**使用场景**:
```
调度实例: schedule_inst_001
- 触发时间: 2025-01-26 12:00:00
- 触发的部署: [1, 2, 3]
- 创建的工作流实例:
  - deployment_id=1 → wf_inst_123456 (成功)
  - deployment_id=2 → wf_inst_123457 (成功)
  - deployment_id=3 → wf_inst_123458 (失败)
- 成功数: 2, 失败数: 1
```

**优势**:
- **完整追踪**: 记录每次调度触发的完整信息
- **统计分析**: 便于统计调度的执行情况
- **问题排查**: 可以查询历史调度记录，定位问题

### 2.10 定义与实例的关系（更新）

| 定义（Definition） | 实例（Instance） | 说明 |
|-------------------|-----------------|------|
| Workflow | WorkflowInstance | 工作流定义 → 工作流实例 |
| Workflow | Deployment | 工作流定义 → 部署对象（封装层） |
| Schedule | ScheduleInstance | 调度配置 → 调度实例 |
| ScheduleInstance | WorkflowInstance | 调度实例 → 工作流实例（1:N） |
| Deployment | WorkflowInstance | 部署对象 → 工作流实例 |
| Node | NodeInstance | 节点定义 → 节点实例 |
| Variable | VariableInstance | 变量定义 → 变量实例 |
| Line | LineInstance | 连接线定义 → 连接线实例 |
| Condition | ConditionInstance | 条件定义 → 条件实例 |

## 3. 功能需求

### 3.1 工作流管理

#### 3.1.1 工作流创建
- 创建新的工作流定义
- 设置工作流基本信息（名称、描述、分类、标签）
- 配置流程结构（节点、连接线、变量）
- 保存流程结构为 JSON 格式
- 流程有多个版本，当前最新发布版本为使用版本，其他版本为历史版本，可以存在一个草稿版本，草稿版本可以发布为正式版本，只有正式版本方可使用
- 新建的工作流默认为草稿，编辑调试完成之后，发布成发布版本，上一个发布版本变成历史版本。版本号由0开始，依次递增，每次加1，不支持回滚到历史版本。

**版本管理规则**:
- 每个工作流（通过 key_name 标识）只能有一个草稿版本（status=draft）
- 可以有多个发布版本（status=published），最新的发布版本为当前使用版本
- 草稿版本可以随时编辑和保存，不影响已发布版本
- 发布草稿时，草稿版本的 status 改为 published，version 设置为当前最大 version + 1
- 发布时如果有正在运行的旧版本实例，不影响其继续执行，新启动的实例使用新版本
- 历史版本（status=archived）仅用于查看和审计，不能再次发布或编辑

#### 3.1.2 工作流编辑
- 修改工作流基本信息
- 添加/删除/修改节点
- 添加/删除/修改连接线
- 配置节点参数和变量映射
- 支持版本管理（保存历史版本）

#### 3.1.3 工作流查询
- 分页查询工作流列表
- 按名称、分类、标签搜索
- 查看工作流详情
- 查看工作流版本历史
- 查看工作流执行统计

#### 3.1.4 工作流删除
- 软删除工作流定义
- 删除前检查是否有运行中的实例
- 支持批量删除

#### 3.1.5 工作流发布
- 将工作流发布到指定盒子
- 支持批量发布到多个盒子
- 验证流程配置的完整性和正确性
- 下发流程 JSON 到盒子端

#### 3.1.6 工作流部署回滚
- 支持回滚到历史版本
- 回滚会创建新版本（内容与目标版本相同）
- 支持灰度发布（分批部署）
- 部署失败自动回滚

**回滚规则**:
- 只能回滚到已发布的历史版本（status=published 或 archived）
- 回滚不会改变版本号，而是创建一个新版本（version = 当前最大 version + 1）
- 新版本的 structure_json 与目标版本完全相同
- 回滚会触发重新部署到指定盒子

**回滚流程**:
1. 验证目标版本存在且状态为 published 或 archived
2. 复制目标版本的 structure_json
3. 创建新版本（version = 当前最大 version + 1，status = published）
4. 将上一个 published 版本标记为 archived
5. 重新部署到指定盒子
6. 记录回滚操作到审计日志

**部署失败处理**:
- 如果部署到某个盒子失败，不影响其他盒子的部署
- 记录失败的盒子 ID 和详细错误信息
- 失败的部署状态标记为 failed
- 支持单独重试失败的部署
- 提供部署状态查询接口

**灰度发布支持**:
- 支持分批部署到不同的盒子组
- 第一批部署到少量盒子（如 10%）进行验证
- 验证无问题后，再部署到其余盒子
- 实现方式：分批调用发布接口，每次指定不同的 box_ids
- 每批部署之间可以设置观察期

**灰度发布流程示例**:
```
第一批（10%）: box_ids=[1, 2]
    ↓ 观察 30 分钟
第二批（50%）: box_ids=[3, 4, 5, 6, 7]
    ↓ 观察 1 小时
第三批（100%）: box_ids=[8, 9, 10, ...]
```

### 3.2 工作流执行

#### 3.2.1 触发方式
- **手动触发**: 用户手动启动工作流执行
- **事件触发**: 基于系统事件自动触发（如：任务完成、盒子上线）
- **定时触发**: 按照 Cron 表达式定时执行
- **API 触发**: 通过 API 接口触发执行

#### 3.2.2 执行控制
- 启动工作流实例
- 暂停工作流实例
- 恢复工作流实例
- 终止工作流实例
- 重试失败的工作流实例

#### 3.2.3 执行监控
- 实时查看工作流实例状态
- 查看当前执行到哪个节点
- 查看节点执行进度
- 查看变量值变化
- 接收执行状态变更通知（SSE）

### 3.3 节点管理

#### 3.3.1 节点模板管理
节点模板定义了系统中可用的节点类型，用户在设计工作流时从节点模板库中拖拽节点到画布。

**节点模板功能**:
- 查询可用的节点模板列表
- 查看节点模板详情（配置参数、输入输出定义）
- 系统预置节点模板（逻辑控制类、业务执行类）
- 支持自定义节点模板（基于 Python 脚本）
- 节点模板分类管理（逻辑控制、业务执行）

**节点模板包含的信息**:
- 节点类型标识（type_key）
- 节点类型名称（type_name）
- 节点分类（category）
- 节点图标（icon）
- 节点描述（description）
- 配置参数定义（config_schema）
- 输入参数定义（input_schema）
- 输出参数定义（output_schema）
- Python 脚本模板（script_template）
- 是否系统预置（is_system）

**系统预置节点模板**:
1. Start - 开始节点
2. End - 结束节点
3. ConcurrencyStart - 并发开始节点
4. ConcurrencyEnd - 并发结束节点
5. LoopStart - 循环开始节点
6. LoopEnd - 循环结束节点
7. KVM - KVM 接入节点
8. Reasoning - AI 推理节点
9. PythonScript - Python 脚本节点
10. MQTT - MQTT 推送节点

**使用流程**:
1. 用户打开工作流设计器
2. 从节点模板库中选择需要的节点类型
3. 拖拽到画布，系统根据模板创建节点实例
4. 节点实例继承模板的配置参数定义
5. 用户填写具体的配置参数值
6. 保存到工作流的 structure_json 中

#### 3.3.2 节点配置
- 配置节点基本信息（名称、描述）
- 配置节点输入参数
- 配置节点输出参数
- 配置节点执行超时时间
- 配置节点失败重试策略
- Python 脚本节点可以编写 Python 脚本，语法验证，测试执行
- 最终节点转化成一段 Python 脚本保存，执行的时候，就是执行这段脚本，下发之后客户端也是运行这段 Python 脚本

#### 3.3.3 节点执行
- 按照流程顺序执行节点
- 支持并发执行多个节点
- 支持循环执行节点
- 记录节点执行日志
- 处理节点执行异常


### 3.4 变量管理

#### 3.4.1 变量定义
- 定义全局变量
- 定义节点变量
- 设置变量类型（字符串、数字、布尔、对象、数组、引用）
- 设置变量默认值
- 设置变量是否必填

#### 3.4.2 变量映射
- 配置节点输入变量来源（上游节点输出、全局变量、常量）
- 配置节点输出变量目标（全局变量、下游节点输入）
- 支持变量类型转换
- 支持变量表达式计算

#### 3.4.3 变量引用语法
**引用格式**: `节点key_name.参数key_name`

**示例**:
- `start_node.image_url` - 引用开始节点的 image_url 参数
- `ai_inference.result` - 引用 AI 推理节点的 result 输出
- `ai_inference.result.confidence` - 引用嵌套对象的属性

**引用规则**:
- 只能引用当前节点之前已执行的节点的输出参数
- 引用的参数必须在被引用节点的 outputs 中定义
- 支持多级属性访问（使用点号分隔）
- 引用不存在的参数会导致执行错误

**类型转换**:
- 自动转换：string ↔ number、boolean ↔ string
- 显式转换：使用内置函数 `toInt()`, `toString()`, `toBool()`
- 对象和数组不支持自动转换

#### 3.5.2 节点日志

#### 3.5.3 日志查询

- 按时间范围查询日志

#### 3.6.4 调度实例管理
- 查询调度实例列表（支持按调度ID、状态、触发类型、时间范围过滤）
- 查看调度实例详情（包含所有关联的工作流实例）
- 查看调度实例统计（成功率、平均耗时、触发类型分布等）
- 实时监控调度实例执行进度（通过 SSE）

**调度实例特性**:
- 每次调度触发创建一个调度实例
- 调度实例记录触发时间、触发类型、触发数据
- 调度实例包含多个工作流实例（每个部署一个）
- 调度实例状态根据工作流实例状态自动更新
- 支持查看调度实例的完整执行链路

**调度实例状态管理**:
- `pending`: 等待执行
- `running`: 执行中（至少有一个工作流实例在运行）
- `completed`: 已完成（所有工作流实例都已完成）
- `failed`: 执行失败（所有工作流实例都失败）
- `cancelled`: 已取消

**调度实例统计分析**:
- 按时间范围统计调度执行情况
- 统计成功率、失败率
- 统计平均执行时长
- 统计触发类型分布
- 支持按日/周/月聚合统计

### 3.7 盒子端集成

#### 3.7.1 工作流下发
- 将工作流 JSON 下发到指定盒子
- 验证盒子是否支持工作流中的节点类型
- 检查盒子资源是否满足执行要求
- 支持批量下发

**下发流程**:
1. 验证工作流配置完整性
2. 检查目标盒子在线状态
3. 验证盒子支持的节点类型
4. 检查盒子资源（CPU、内存、磁盘、已部署模型）
5. 调用盒子端部署 API
6. 等待盒子端确认部署结果
7. 更新 workflow_deployments 表状态

**失败处理**:
- 如果验证失败，返回具体错误信息，不进行下发
- 如果下发失败，标记 deployment_status 为 failed，记录错误信息
- 支持重新下发失败的工作流

#### 3.7.2 执行状态同步
- 盒子端定期上报工作流实例状态
- 盒子端上报节点执行进度
- 盒子端上报执行日志
- 服务端实时更新状态

**同步机制**:
- 盒子端每 5 秒主动上报一次状态（心跳）
- 节点状态变更时立即上报（事件驱动）
- 服务端接收后更新数据库
- 通过 SSE 推送状态变更给前端

**断线重连**:
- 盒子端断线后，保留本地执行状态
- 重连后，上报所有运行中实例的最新状态
- 服务端根据上报数据更新状态
- 如果状态冲突，以盒子端状态为准

#### 3.7.3 远程控制
- 服务端远程启动盒子上的工作流
- 服务端远程暂停/恢复/终止工作流
- 服务端查询盒子上的工作流状态

**控制流程**:
1. 服务端调用盒子端控制 API
2. 盒子端执行控制命令
3. 盒子端返回执行结果
4. 服务端更新实例状态
5. 通过 SSE 通知前端

**超时处理**:
- 控制命令超时时间：30 秒
- 超时后标记命令失败
- 支持重试（最多 3 次）

#### 4.0.3 数据流转

1. **设计阶段**:
   - 用户从 `node_templates` 选择节点模板
   - 前端发送创建工作流请求，包含 structure 对象
   - 后端保存到 `workflows` 表（基本信息 + structure_json）
   - 后端保存到 `node_definitions` 表（节点列表）
   - 后端保存到 `line_definitions` 表（连接线列表）
   - 后端保存到 `variable_definitions` 表（变量列表）
   - 使用事务保证数据一致性

2. **查询阶段**:
   - 前端请求工作流详情
   - 后端查询 `workflows` 表
   - 后端可选择：
     - 方式1：直接返回 structure_json 字段（快速）
     - 方式2：关联查询 node_definitions、line_definitions、variable_definitions 组装（灵活）
   - 推荐编辑时使用方式2，查看时使用方式1

3. **部署阶段**:
   - 直接读取 workflows.structure_json 字段
   - 将 JSON 下发到盒子（无需关联查询，性能更好）
   - 创建 `workflow_deployments` 记录

4. **执行阶段**:
   - 创建 `workflow_instances` 记录
   - 基于 `node_definitions` 创建 `node_instances` 记录
   - 基于 `variable_definitions` 创建 `variable_instances` 记录
   - 基于 `line_definitions` 创建 `line_instances` 记录
   - 记录 `workflow_logs`

2. **AI推理节点**: 使用YOLOv5模型进行目标检测
3. **Python脚本节点**: 根据置信度阈值过滤结果
4. **条件分支**: 
   - 如果有有效结果 → 进入并发处理
   - 如果无有效结果 → 记录日志后结束
5. **并发处理**: 同时执行两个任务
   - 推送结果到MQTT服务器
   - 保存结果到数据库
6. **并发结束**: 等待两个并发任务完成
7. **结束节点**: 工作流结束

**流程图**:
```
Start → AI推理 → 结果过滤 → 条件判断
                              ├─ 有结果 → 并发开始 ┬─ MQTT推送 ─┬─ 并发结束 → End
                              │                   └─ 保存数据库 ─┘
                              └─ 无结果 → 记录日志 → End
```

**使用示例**:
```json

}

// OR 逻辑（任意条件满足即可）
{
  "condition_type": "complex",
  "logic_type": "or",
  "condition_expression": "result.error != ''; result.timeout == true"
}
```

**script_template 示例（Reasoning 节点）**:
```python
def execute(context, config, inputs):
    """
    AI 推理节点执行脚本
    
    Args:
        context: 工作流上下文对象
        config: 节点配置参数
        inputs: 节点输入参数
    
    Returns:
        dict: 节点输出结果
    """

    timeout = config.get('timeout', 30)
```

### 4.12 节点定义表（node_definitions）

节点定义表存储工作流中实际使用的节点配置，是从节点模板实例化而来的。

| 字段名 | 类型 | 说明 | 约束 |
|-------|------|------|------|
| id | uint | 主键ID | PK, Auto Increment |
| workflow_id | uint | 工作流ID | NOT NULL, FK |

| node_template_id | uint | 节点模板ID | NOT NULL, FK |
| type_key | string | 节点类型标识 | NOT NULL |
| type_name | string | 节点类型名称 | NOT NULL |

| outputs | jsonb | 输出参数定义 | - |
| position | jsonb | 节点在画布上的位置 | - |
| created_at | timestamp | 创建时间 | NOT NULL |
| updated_at | timestamp | 更新时间 | NOT NULL |

**索引**:
- `idx_workflow_id` (workflow_id)
- `idx_node_id` (node_id)
- `idx_node_template_id` (node_template_id)
- `idx_node_key_name` (node_key_name)

**唯一约束**:
- `unique_workflow_node` (workflow_id, node_id)
- `unique_workflow_key_name` (workflow_id, node_key_name)

**说明**:
- `node_definitions` 表存储工作流中实际使用的节点
- 每个节点定义都基于一个节点模板（node_template_id）
- `config` 字段存储用户填写的具体配置值

#### 4.13.4 验证规则

工作流 JSON 需要满足以下验证规则：

1. **节点验证**:
   - 必须有且仅有一个 start 节点
   - 必须有且仅有一个 end 节点
   - 所有节点的 id 必须唯一
   - 所有节点的 key_name 必须唯一
   - 成对节点必须配对出现（如 concurrency_start 和 concurrency_end）

2. **连接线验证**:
   - source 和 target 必须指向存在的节点
   - 不能有循环依赖（除了循环节点）
   - 所有节点（除 end 节点）必须至少有一条输出连接线
   - 所有节点（除 start 节点）必须至少有一条输入连接线

3. **变量验证**:
   - 全局变量的 key_name 必须唯一
   - 引用类型的参数必须指向存在的节点输出

4. **条件验证**:
   - 条件表达式语法必须正确
   - 条件中引用的变量必须存在

  5. 组装成完整的 structure 对象返回
- 推荐：查询详情时使用方式2（保证数据最新），下发给盒子时使用方式1（性能更好）

### 6.1 工作流执行引擎

#### 6.1.1 执行流程
1. **初始化阶段**
   - 创建工作流实例
   - 初始化全局变量
   - 创建节点实例
   - 创建连接线实例

2. **执行阶段**
   - 从 Start 节点开始
   - 按照连接线顺序执行节点
   - 处理节点输入输出
   - 更新变量值
   - 记录执行日志

3. **完成阶段**
   - 到达 End 节点
   - 更新工作流实例状态
   - 计算执行耗时
   - 触发完成事件

#### 6.1.2 节点执行器
每种节点类型对应一个执行器（Executor），负责具体的执行逻辑。

**说明**:
- 所有节点最终都转化为 Python 脚本执行
- 执行器负责调用 Python 解释器执行脚本
- 脚本执行在沙箱环境中，限制权限

**逻辑控制类执行器**:
- `StartExecutor`: 初始化执行上下文
- `EndExecutor`: 结束执行，返回结果
- `ConcurrencyStartExecutor`: 创建并发执行分支
- `ConcurrencyEndExecutor`: 等待所有并发分支完成
- `LoopStartExecutor`: 初始化循环上下文
- `LoopEndExecutor`: 判断循环条件，决定是否继续

**业务执行类执行器**:
- `KVMExecutor`: 调用 KVM 接入服务
- `ReasoningExecutor`: 调用 AI 推理服务
- `PythonScriptExecutor`: 执行 Python 脚本
- `MQTTExecutor`: 推送 MQTT 消息

#### 6.1.3 条件评估器
负责评估连接线上的条件表达式：

**支持的运算符**:
- 比较运算符: `==`, `!=`, `>`, `<`, `>=`, `<=`
- 逻辑运算符: `&&`, `||`, `!`
- 成员运算符: `in`, `not in`
- 正则匹配: `matches`

**表达式示例**:
```javascript
// 简单条件
result.confidence > 0.8

// 复合条件
result.confidence > 0.8 && result.class == "person"

// 成员判断
result.class in ["person", "car", "bicycle"]

// 正则匹配
result.message matches "^success"
```

#### 6.1.4 变量管理器
负责管理工作流执行过程中的变量：

**功能**:
- 变量定义和初始化
- 变量值的读取和写入
- 变量作用域管理
- 变量类型转换
- 变量引用解析（如：`node_a.result`）

**Cron 表达式示例**:

**实现方式**:

#### 6.2.3 并发控制
控制同一工作流的并发执行数量：

**策略**:
- 检查当前运行中的实例数
- 如果达到 `max_concurrent` 限制，则排队等待
- 使用优先级队列管理等待的实例

### 6.3 盒子端集成

#### 6.3.1 部署和调度下发流程

**架构关系**:
```
Workflow (工作流定义)
    ↓ 1:N
Deployment (部署实例) ← 关联 Box
    ↓ 1:N
Schedule (调度配置)
    ↓ 触发
WorkflowInstance (执行实例)
```

**完整下发流程**:

1. **部署实例下发**
   - 服务端创建部署实例（deployment）
   - 部署实例包含：workflow 定义 + 参数配置 + 关联的盒子
   - 调用盒子端 `/api/deployments/receive` 接口下发部署实例
   - 盒子端保存部署实例，准备接收调度配置
   
   **下发数据**:
   ```json
   {
     "deployment_id": 1,
     "deployment_key": "video_analysis_camera1",
     "deployment_name": "摄像头1视频分析",
     "workflow_id": 1,
     "workflow_json": { /* workflow 完整定义 */ },
     "variables": [ /* 部署级别的参数配置 */ ]
   }
   ```

2. **调度配置下发**
   - 服务端创建调度配置（schedule）
   - 调度配置关联到部署实例（deployment_id）
   - 调用盒子端 `/api/schedules/receive` 接口下发调度配置
   - 盒子端启动调度器，按配置触发部署实例执行
   
   **下发数据**:
   ```json
   {
     "schedule_id": 1,
     "schedule_name": "每小时分析一次",
     "deployment_id": 1,
     "deployment_key": "video_analysis_camera1",
     "schedule_type": "cron",
     "cron_expression": "0 * * * *",
     "is_enabled": true,
     "max_concurrent": 1,
     "timeout": 3600
   }
   ```

3. **验证阶段**
   - 检查盒子是否在线
   - 验证盒子是否支持工作流中的所有节点类型
   - 检查盒子资源是否满足要求（CPU、内存、模型）
   - 验证调度配置的合法性（cron 表达式、并发限制等）

4. **确认阶段**
   - 更新 `workflow_deployments` 表状态为 deployed
   - 更新 `workflow_schedules` 表状态为 enabled
   - 记录部署和调度日志
   - 发送下发成功/失败通知

**下发顺序**:
- 必须先下发部署实例，再下发调度配置
- 调度配置依赖部署实例（通过 deployment_id 关联）
- 如果部署实例下发失败，不进行调度配置下发

**失败处理**:
- 如果验证失败，返回具体错误信息，不进行下发
- 如果部署实例下发失败，标记 deployment_status 为 failed
- 如果调度配置下发失败，标记 schedule_status 为 failed
- 支持重新下发失败的部署和调度

**更新场景**:
- 如果部署实例已存在，下发会更新现有部署
- 如果调度配置已存在，下发会更新现有调度
- 更新时会先停止旧的调度，再启动新的调度
- 支持重新下发失败的部署和调度

#### 6.3.2 执行状态同步

**同步架构**:
```
调度触发 → 部署执行 → Workflow 执行 → 节点执行
    ↓          ↓           ↓            ↓
调度状态                 实例状态      节点状态
    ↓          ↓           ↓            ↓
        实时同步到服务端（带重试机制）
```

**盒子端 → 服务端**:
- 盒子端实时上报各个实体的执行状态
- 上报内容包括：调度状态、工作流实例状态、节点实例状态、执行日志
- 服务端更新 `workflow_schedules`、`workflow_deployments`、`workflow_instances`、`node_instances` 和 `workflow_logs` 表
- 所有记录都包含 schedule_id 和 deployment_id，便于追踪

**同步 API**:
```
POST /api/v1/box-client/schedules/{schedule_id}/sync
```

**请求体**:
```json
{
  "schedule_id": 1,
  "deployment_id": 1,
  "schedule_status": "enabled",
  "last_run_time": "2025-01-26T12:00:00Z",
  "next_run_time": "2025-01-26T13:00:00Z",
  "workflow_instances": [
    {
      "instance_id": "wf_inst_123456",
      "workflow_id": 1,
      "status": "running",
      "current_node_id": "node_2",
      "start_time": "2025-01-26T12:00:00Z",
      "variables": {
        "result": {"confidence": 0.95, "class": "person"}
      },
      "node_instances": [
        {
          "instance_id": "node_inst_123",
          "node_id": "node_2",
          "status": "running",
          "start_time": "2025-01-26T12:01:00Z"
        }
      ],
      "logs": [
        {
          "log_type": "node",
          "operation_instance_id": "node_inst_123",
          "operation_instance_name": "AI推理",
          "message": "节点开始执行",
          "created_at": "2025-01-26T12:01:00Z"
        }
      ]
    }
  ]
}
```

**同步机制**:

1. **实时同步（事件驱动）**:
   - 每次节点状态变更时立即上报
   - 每次工作流实例状态变更时立即上报
   - 每次产生执行日志时立即上报
   - 确保服务端能实时获取最新状态

2. **心跳同步（定时）**:
   - 盒子端每 5 秒主动上报一次完整状态（心跳）
   - 包含调度状态、所有运行中的实例、最新的日志
   - 作为实时同步的补充，防止消息丢失

3. **重试机制**:
   - 如果同步请求失败（网络错误、服务端错误等），将同步数据加入重试队列
   - 重试队列由独立线程处理，每隔 5 秒重新发送一次
   - 同步成功后从队列中移除
   - 最多重试 3 次，超过后记录错误日志并丢弃
   - 重试队列有容量限制（如 1000 条），防止内存溢出

4. **数据合并**:
   - 重试队列中相同实例的多次更新会被合并
   - 只保留最新的状态数据
   - 减少重试次数，提高效率

**重试队列实现**:
```go
type SyncQueue struct {
    queue    chan *SyncData
    maxRetry int
    interval time.Duration
}

type SyncData struct {
    ScheduleID   uint
    DeploymentID uint
    Data         interface{}
    RetryCount   int
    LastRetry    time.Time
}

// 同步失败时加入队列
func (q *SyncQueue) Enqueue(data *SyncData) {
    select {
    case q.queue <- data:
        // 成功加入队列
    default:
        // 队列已满，记录错误日志
        log.Error("Sync queue is full, dropping data")
    }
}

// 重试线程
func (q *SyncQueue) RetryWorker() {
    ticker := time.NewTicker(q.interval)
    for {
        select {
        case <-ticker.C:
            q.processRetry()
        }
    }
}
```

**服务端处理**:
- 服务端接收后更新数据库
- 使用事务保证数据一致性
- 更新成功后返回 200 状态码
- 更新失败返回错误信息，触发盒子端重试
- 通过 SSE 推送状态变更给前端

**断线重连**:
- 盒子端断线后，保留本地执行状态和未同步的数据
- 重连后，立即上报所有运行中实例的最新状态
- 重连后，处理重试队列中的未同步数据
- 服务端根据上报数据更新状态
- 如果状态冲突，以盒子端状态为准（盒子端是真实执行环境）

**性能优化**:
- 批量上报：一次请求包含多个实例的状态
- 增量上报：只上报变更的数据，不上报未变更的数据
- 压缩传输：对大数据量使用 gzip 压缩
- 异步处理：同步操作不阻塞业务执行

#### 6.3.3 远程控制

**服务端 → 盒子端**:
- 服务端通过调度 ID 向盒子发送控制命令
- 支持的命令：启动、暂停、恢复、停止
- 盒子端根据调度配置执行对应的部署实例
- 盒子端执行命令并返回结果

**控制 API（盒子端提供）**:
```
POST /api/schedules/{schedule_id}/start    # 启动调度（创建新的工作流实例）
POST /api/schedules/{schedule_id}/pause    # 暂停调度（停止创建新实例）
POST /api/schedules/{schedule_id}/resume   # 恢复调度
POST /api/schedules/{schedule_id}/stop     # 停止调度
```

**控制流程**:
1. 服务端调用盒子端调度控制 API
2. 盒子端根据调度配置找到关联的部署实例
3. 盒子端执行控制命令（如启动调度会创建新的工作流实例）
4. 盒子端返回执行结果
5. 服务端更新调度和实例状态
6. 通过 SSE 通知前端

**超时处理**:
- 控制命令超时时间：30 秒
- 超时后标记命令失败
- 支持重试（最多 3 次）

**架构优势**:
- 调度和部署分离，职责清晰
- 同一 workflow 可以有多个部署实例（不同参数）
- 每个部署实例可以有独立的调度策略
- 便于管理和监控（通过 schedule_id 和 deployment_id 追踪）


### 6.4 错误处理和重试

#### 6.4.1 节点执行失败处理
1. **记录错误信息**
   - 更新节点实例状态为 `failed`
   - 记录错误消息和堆栈信息
   - 写入日志

2. **重试策略**
   - 检查节点配置的重试次数
   - 如果未达到最大重试次数，则重试
   - 重试间隔：指数退避（1s, 2s, 4s, 8s...）

3. **失败传播**
   - 如果重试仍然失败，标记节点为最终失败
   - 根据配置决定是否继续执行后续节点
   - 或者标记整个工作流实例为失败

#### 6.4.2 工作流实例失败处理
1. **失败通知**
   - 发送 SSE 事件通知前端
   - 发送系统告警（如果配置）
   - 记录失败日志

2. **失败恢复**
   - 支持从失败节点重新执行
   - 保留之前成功节点的输出
   - 重新初始化失败节点的输入

#### 6.4.3 超时处理
1. **节点超时**
   - 每个节点可配置超时时间
   - 超时后自动终止节点执行
   - 标记节点为超时失败

2. **工作流超时**
   - 整个工作流可配置总超时时间
   - 超时后终止所有运行中的节点
   - 标记工作流实例为超时失败

### 6.5 并发和循环控制

#### 6.5.1 并发执行
**实现方式**:
1. 遇到 `ConcurrencyStart` 节点时
2. 创建多个并发分支（goroutine）
3. 每个分支独立执行后续节点
4. 到达 `ConcurrencyEnd` 节点时等待所有分支完成
5. 合并各分支的输出结果

**并发控制**:
- 使用 `sync.WaitGroup` 等待所有分支
- 使用 channel 收集各分支结果
- 支持配置最大并发数

**异常处理**:
- **策略一：快速失败（默认）**
  - 任一分支失败，立即取消其他分支
  - 标记整个并发区域为失败
  - 工作流实例状态变为 failed
  
- **策略二：继续执行**
  - 某个分支失败，不影响其他分支
  - 记录失败分支的错误信息
  - 等待所有分支完成后，根据失败数量决定整体状态
  - 如果所有分支都失败，标记为失败
  - 如果部分分支成功，标记为部分成功（partial_success）

- **策略三：最少成功数**
  - 配置最少需要成功的分支数（如：3 个分支至少 2 个成功）
  - 达到最少成功数后，可以提前结束等待
  - 未达到最少成功数，标记为失败

**并发配置示例**:
```json
{
  "id": "concurrency_start_1",
  "type": "concurrency_start",
  "config": {
    "max_concurrent": 5,
    "failure_strategy": "fast_fail",
    "min_success_count": 0,
    "timeout": 300
  }
}
```

**变量作用域**:
- 每个并发分支有独立的变量作用域
- 分支可以读取全局变量，但修改不会影响其他分支
- 分支的输出变量会合并到全局变量（使用分支索引作为前缀，如：`branch_0.result`）

**并发结果合并**:
```json
{
  "concurrency_results": [
    {
      "branch_index": 0,
      "status": "completed",
      "output": {"result": "success"}
    },
    {
      "branch_index": 1,
      "status": "completed",
      "output": {"result": "success"}
    },
    {
      "branch_index": 2,
      "status": "failed",
      "error": "timeout"
    }
  ],
  "success_count": 2,
  "failed_count": 1,
  "total_count": 3
}
```

#### 6.5.2 循环执行
**实现方式**:
1. 遇到 `LoopStart` 节点时
2. 初始化循环计数器和循环变量
3. 执行循环体内的节点
4. 到达 `LoopEnd` 节点时评估循环条件
5. 如果条件满足，跳回 `LoopStart` 继续
6. 如果条件不满足或达到最大循环次数，退出循环

**循环控制**:
- 支持配置最大循环次数（防止死循环）
- 支持 `break` 和 `continue` 语义
- 循环变量在每次迭代中更新

**循环类型**:

**1. 条件循环（while）**:
```json
{
  "id": "loop_start_1",
  "type": "loop_start",
  "config": {
    "loop_type": "while",
    "condition": "task_count > 0",
    "max_iterations": 100
  }
}
```

**2. 计数循环（for）**:
```json
{
  "id": "loop_start_2",
  "type": "loop_start",
  "config": {
    "loop_type": "for",
    "loop_variable": "i",
    "start": 0,
    "end": 10,
    "step": 1
  }
}
```

**3. 集合循环（foreach）**:
```json
{
  "id": "loop_start_3",
  "type": "loop_start",
  "config": {
    "loop_type": "foreach",
    "loop_variable": "item",
    "collection": "task_list"
  }
}
```

**循环变量**:
- 循环变量在循环体内可见
- 每次迭代更新循环变量的值
- 循环变量命名空间：`loop.{loop_variable}`
- 访问示例：`loop.i`, `loop.item`

**特殊循环变量**:
- `loop.index` - 当前迭代索引（从 0 开始）
- `loop.count` - 当前迭代次数（从 1 开始）
- `loop.first` - 是否第一次迭代（布尔值）
- `loop.last` - 是否最后一次迭代（布尔值，仅 for 和 foreach）

**循环控制语句**:
- **break**: 立即退出循环，执行 LoopEnd 之后的节点
- **continue**: 跳过本次迭代，继续下一次迭代

实现方式：在循环体内的节点输出中设置特殊标记：
```json
{
  "output": {
    "result": "...",
    "__loop_control__": "break"  // 或 "continue"
  }
}
```

**嵌套循环**:
- 支持循环嵌套（最多 3 层）
- 内层循环的变量不会影响外层循环
- 循环变量使用完全限定名：`loop.outer.i`, `loop.inner.j`

**循环超时**:
- 支持配置整个循环的超时时间
- 超时后自动退出循环，标记为超时失败
- 已完成的迭代结果会保留

**循环结果收集**:
```json
{
  "loop_results": [
    {
      "iteration": 0,
      "loop_variable": {"i": 0},
      "output": {"result": "success"}
    },
    {
      "iteration": 1,
      "loop_variable": {"i": 1},
      "output": {"result": "success"}
    }
  ],
  "total_iterations": 2,
  "success_count": 2,
  "failed_count": 0
}
```

**循环性能优化**:
- 循环体内的节点实例复用（避免重复创建）
- 循环变量增量更新（避免全量复制）
- 循环结果流式收集（避免内存溢出）

## 7. 非功能需求

### 7.1 性能要求

#### 7.1.1 响应时间
- API 接口响应时间 < 200ms（P95）
- 工作流启动时间 < 1s
- 节点执行延迟 < 100ms（不含业务逻辑耗时）
- 状态同步延迟 < 5s

#### 7.1.2 吞吐量
- 支持单服务器 1000+ 并发工作流实例
- 支持单服务器 10000+ 并发节点执行
- 支持每秒 100+ 工作流启动请求
- 支持每秒 1000+ 状态同步请求

#### 7.1.3 资源占用
- 单个工作流实例内存占用 < 10MB
- 单个节点实例内存占用 < 1MB

#### 7.3.2 数据安全

- 盒子端通信使用 TLS
- 支持数据脱敏（日志中的敏感信息）

**敏感数据处理**:
- 变量值中的敏感数据（如密码、API Key）使用 AES-256 加密存储
- 日志中自动脱敏敏感字段（如：password、token、secret、key）
- 脱敏规则：保留前 2 位和后 2 位，中间用 * 替代（如：ab****xy）

#### 7.3.3 代码安全
- 防止 SQL 注入（使用参数化查询）
- 防止 XSS 攻击（输入验证和输出转义）
- 防止 CSRF 攻击（使用 CSRF Token）
- 限制 Python 脚本执行权限（沙箱环境）

**Python 脚本安全限制**:

**禁用模块列表**:
```python
# 文件系统操作
os, sys, subprocess, shutil, pathlib, glob

# 网络操作（除了允许的 HTTP 客户端）
socket, urllib, urllib3, requests (仅允许通过封装的安全 HTTP 客户端)

# 代码执行
eval, exec, compile, __import__, importlib

# 系统调用
ctypes, cffi, pty, fcntl

# 多进程/多线程
multiprocessing, threading (仅允许有限的异步操作)
```

**允许的模块列表**:
```python
# 数据处理
json, re, datetime, time, math, random, decimal, uuid

# 数据结构
collections, itertools, functools

# 字符串处理
string, textwrap

# 类型检查
typing, dataclasses

# 自定义封装的安全模块
safe_http (封装的 HTTP 客户端，仅允许访问白名单域名)
safe_log (封装的日志模块)
workflow_context (工作流上下文，访问变量和节点输出)
```

**资源限制**:
- CPU 时间限制：单个脚本最多执行 60 秒
- 内存限制：单个脚本最多使用 256MB 内存
- 输出大小限制：单个脚本输出最多 10MB
- 递归深度限制：最大递归深度 100

**沙箱实现方式**:
1. 使用 RestrictedPython 库限制 Python 语法
2. 自定义 `__builtins__` 字典，只包含安全的内置函数
3. 使用 `resource` 模块限制 CPU 和内存使用
4. 在独立的进程中执行脚本，通过 IPC 通信
5. 设置进程超时，超时后强制终止

**脚本执行示例**:
```python
# 用户编写的脚本

    # 处理数据
    result = {

    }
    
    # 返回结果
    return result
```

```go
type WorkflowExecutorService interface {
```

## 10. 使用场景示例

### 10.1 场景一：视频 AI 分析工作流

**业务需求**:
对上传的视频进行 AI 分析，识别视频中的人员和车辆，并将结果推送到 MQTT 服务器。

**工作流设计**:
```
[Start] → [抽帧] → [AI推理] → [结果过滤] → [MQTT推送] → [End]
                                    ↓ (confidence < 0.8)
                                [记录日志] → [End]
```

**节点配置**:
1. **Start 节点**: 接收视频 URL 作为输入
2. **抽帧节点**: 每秒抽取 1 帧
3. **AI 推理节点**: 使用 YOLOv5 模型进行目标检测
4. **结果过滤节点**: 根据置信度过滤结果
5. **MQTT 推送节点**: 推送到指定 Topic
6. **记录日志节点**: 记录低置信度结果
7. **End 节点**: 结束流程

**触发方式**: 手动触发或 API 触发

### 10.2 场景二：定时巡检工作流

**业务需求**:
每天凌晨 2 点自动检查所有在线盒子的健康状态，如果发现异常则发送告警。

**工作流设计**:
```
[Start] → [获取盒子列表] → [并发开始]
                              ↓
                    [检查盒子1] [检查盒子2] [检查盒子3] ...
                              ↓
                          [并发结束]
                              ↓
                        [汇总结果]
                              ↓
                    [是否有异常?] → (是) → [发送告警] → [End]
                              ↓ (否)
                            [End]
```

**节点配置**:
1. **Start 节点**: 无输入
2. **获取盒子列表节点**: 调用盒子管理 API
3. **并发开始节点**: 为每个盒子创建并发分支
4. **检查盒子节点**: 检查盒子 CPU、内存、磁盘使用率
5. **并发结束节点**: 等待所有检查完成
6. **汇总结果节点**: 统计异常盒子数量
7. **条件判断**: 如果有异常则发送告警
8. **发送告警节点**: 调用告警服务
9. **End 节点**: 结束流程

**触发方式**: Cron 表达式 `0 2 * * *`（每天凌晨 2 点）

### 10.3 场景三：数据处理循环工作流

**业务需求**:
批量处理待处理的任务队列，每次处理 10 个任务，直到队列为空。

**工作流设计**:
```
[Start] → [循环开始]
              ↓
        [获取待处理任务]
              ↓
        [是否有任务?] → (否) → [循环结束] → [End]
              ↓ (是)
        [处理任务]
              ↓
        [更新任务状态]
              ↓
        [循环结束] → (继续循环)
```

**节点配置**:
1. **Start 节点**: 无输入
2. **循环开始节点**: 设置最大循环次数 100
3. **获取待处理任务节点**: 从数据库获取 10 个待处理任务
4. **条件判断**: 检查是否有任务
5. **处理任务节点**: 执行业务逻辑
6. **更新任务状态节点**: 标记任务为已完成
7. **循环结束节点**: 判断是否继续循环
8. **End 节点**: 结束流程

**触发方式**: 手动触发或 API 触发

### 11.5 常见问题解答（FAQ）

#### Q1: 工作流的草稿版本和发布版本有什么区别？
**A**: 
- 草稿版本（draft）：可以随时编辑和保存，不会影响已部署的工作流，每个工作流只能有一个草稿版本
- 发布版本（published）：已经部署到盒子端，正在使用的版本，不能编辑，可以有多个历史发布版本
- 发布时，草稿版本会变成新的发布版本，版本号自动递增

#### Q2: 如何引用其他节点的输出参数？
**A**: 使用格式 `节点key_name.参数key_name`，例如：
- `start_node.image_url` - 引用开始节点的 image_url 参数
- `ai_inference.result.confidence` - 引用 AI 推理节点输出的嵌套属性
- 只能引用当前节点之前已执行的节点的输出

#### Q3: 并发节点中某个分支失败了怎么办？
**A**: 有三种策略：
- 快速失败（默认）：任一分支失败，立即取消其他分支
- 继续执行：某个分支失败不影响其他分支，等待所有分支完成
- 最少成功数：配置最少需要成功的分支数，达到即可

#### Q4: 如何防止循环节点死循环？
**A**: 
- 配置最大循环次数（默认 100 次）
- 配置循环超时时间
- 在循环体内使用 break 语句提前退出
- 循环条件应该有明确的终止条件

#### Q5: Python 脚本节点可以做什么？
**A**: 
- 可以执行自定义的数据处理逻辑
- 可以调用封装好的安全 HTTP 客户端
- 可以访问工作流上下文和变量
- 但受到安全限制：禁止文件操作、网络操作、系统调用等

#### Q6: 工作流部署失败了怎么办？
**A**: 
- 查看部署错误信息，确认失败原因
- 使用重试接口单独重试失败的盒子
- 如果是配置问题，修改工作流后重新发布
- 如果是盒子问题，检查盒子状态和资源

#### Q7: 如何实现灰度发布？
**A**: 
- 分批调用发布接口，每次指定不同的 box_ids
- 第一批部署到少量盒子（如 10%）
- 观察一段时间，确认无问题
- 再部署到其余盒子

#### Q8: 工作流实例可以暂停和恢复吗？
**A**: 
- 可以，调用暂停接口后，当前节点执行完成后暂停
- 恢复时从暂停的位置继续执行
- 暂停期间的变量值会保留

#### Q9: 如何查看工作流的执行日志？
**A**: 
- 调用日志查询接口，按工作流实例 ID 查询
- 可以按日志类型过滤（node/line）
- 可以按时间范围查询
- 支持导出日志

#### Q10: 节点执行失败会自动重试吗？
**A**: 
- 会，根据节点配置的重试策略自动重试
- 默认最多重试 3 次
- 重试间隔采用指数退避（1s, 2s, 4s, 8s...）
- 如果重试仍然失败，标记节点为最终失败

#### Q11: 如何回滚到历史版本？
**A**: 
- 调用回滚接口，指定目标版本号
- 回滚会创建一个新版本（内容与目标版本相同）
- 不会真正改变版本号，而是递增版本号
- 回滚后会自动重新部署到指定盒子

#### Q12: 变量的作用域是什么？
**A**: 
- 全局变量：整个工作流实例共享，start 节点的入参为全局变量
- 节点变量：仅在当前节点可见
- 并发分支变量：每个分支有独立的作用域
- 循环变量：在循环体内可见

#### Q13: 如何保证工作流执行的安全性？
**A**: 
- 所有 API 需要 JWT Token 认证
- ~~基于 RBAC 的权限控制~~（暂不实现）
- Python 脚本在沙箱环境中执行
- 敏感数据加密存储
- 日志自动脱敏

#### Q14: 盒子端断线后工作流会怎样？
**A**: 
- 盒子端会保留本地执行状态
- 重连后自动上报最新状态
- 服务端根据上报数据更新状态
- 如果状态冲突，以盒子端状态为准

#### Q15: 如何监控工作流的执行情况？
**A**: 
- 通过 SSE 接口订阅实时事件
- 查询工作流实例状态和进度
- 查看统计分析数据
- 配置 Prometheus 监控指标
- 设置告警规则

#### Q16: 调度实例和工作流实例有什么区别？
**A**:
- 调度实例：记录每次调度触发的信息，一个调度实例可以包含多个工作流实例
- 工作流实例：记录单个工作流的执行情况
- 关系：调度实例 1:N 工作流实例
- 场景：一个调度配置了3个部署，触发一次会创建1个调度实例和3个工作流实例

#### Q17: 如何查看某次调度触发的所有工作流执行情况？
**A**:
- 调用调度实例详情接口 `GET /api/v1/workflow-schedule-instances/{instance_id}`
- 返回结果包含所有关联的工作流实例
- 可以看到每个部署对应的工作流实例状态
- 包含成功数、失败数等统计信息

#### Q18: 如何统计调度的执行情况？
**A**:
- 调用统计接口 `GET /api/v1/workflow-schedule-instances/statistics`
- 可以按时间范围统计
- 包含成功率、失败率、平均耗时
- 包含触发类型分布（cron/manual）
- 支持按日/周/月聚合统计

#### Q19: 一个调度可以配置多少个部署？
**A**:
- 理论上没有限制
- 建议不超过10个部署
- 部署过多会影响调度触发性能
- 可以根据实际需求调整
- 如果需要更多部署，建议创建多个调度配置
